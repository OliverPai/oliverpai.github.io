提交

  • 在准备用作仓库的文件夹中git init

    若无初始化，git status时会提示当前文件夹不是仓库

  • 提交：

点击vscode的这个图标，会出现两个目录：更改和暂存的更改。改动了源码但是add执行前会放入“更改”，add后进入暂存的更改。

  • 查看历史：git log —stat

    每一次commit都会产生一个哈希值（commit号），用来标志这次提交（如果只想查看刚刚commit的状态，可以通过git show命令完成）

    同样地，vscode也有图形化处理，在commit之后就会出现“commit”目录，里面有每一次commit：

    一个文件夹中往往并不是所有文件都要被追踪更改、被加入暂存区。比如深度学习的项目中会有很多图片文件做训练集，它们是没必要被追踪的。这时可以利用.gitignore文件，其负责告诉git有哪些文件不必被追踪。如执行nano .gitignore后，添加

    那么文件夹中所有png文件都不会被追踪。（前提是.gitignore文件已被commit过）

  • 删除文件

    当想在文件夹中删除文件时，不要简单地使用rm命令，因为会导致工作区与暂存区不一致。所以需要：用命令git rm删掉，并且git commit。

    如果已经rm了，依然可以继续git rm和git commit，只要有git rm就不会出问题。

回滚

  • git版CTRL-Z：

    git checkout <file>等同于上面的第二个符号，用于在add前的回滚（工作区回滚，回到上一个版本）

如果已经commit了还要回滚，需要用git reset HEAD^1来进行回滚。HEAD表示当前分支上最近一次提交记录，即当前的commit。若要回到上一次commit，用HEAD1，其中1表示前一次的commit。

→

vscode中实现reset，当前在code里打开某文件。点击commit中的对应commit，右键undo commit。之后文件从仓库回到暂存区，点击减号退出暂存区，再到“更改”目录点击工作区回滚对应符号，对相应文件的回滚完成。

如果想回滚到半年前的版本，自己的记忆又没有那么深刻，那么可以利用vscode的file history，上面具有commit的时间信息，点击对应的commit可以进行文件的今昔对比。

分支

📌master主分支应该是其他各分支的基础，所以在执行git checkout -b <branchname>时，当前分支应该提前切换到master分支，以主分支为根建分支。
比如a和b都是以master为基础新建的分支，那么修改a分支和b分支时，二者互不影响。

但是在合并分支时很有可能出现合并冲突的情况，比如a分支和b分支都以master为基础，均做了修改，a向master合并了，这时b再合并，a和b就有了冲突的代码，vscode的源代码管理也会出现“合并更改”目录，包含存在冲突的文件。

这时候会有选项，Accept Current Change之类的，就可以选择是保存哪个分支的修改，抑或两个分支的修改都保留。

命令行实现上面的功能，用git diff <filename> <branch1> <branch2>查看不同分支对某一文件的修改区别。

如果涉及冲突的文件特别多，那么vscode提供选项，可选全部采用哪种版本的修改。

当遇到自己没法判断的冲突时，需要git merge --abort放弃合并，再找能够解决问题的同事来分析。

  • 查看分支历史：git graph

    比如下面可以看到当前处于main分支，此外还有一个feature分支，在feature分支上做了两次提交：commit5和commit6。

  • merge vs rebase（变基）：

    以下图为例，有两个分支：main和my_feature。my_feature建立分支后有三个commit，同时main也有了三个新commit。如果执行merge命令，git会尝试将两个分支的最新commit进行合并；而rebase的话（git switch my_feature; git rebase main），会执行一次回放操作：把my_feature的三个commit，尝试合并到main最新commit，抹除my_feature分支。

    • merge的问题：加入了很多没有太大意义的merge commit，显得结构混乱，而rebase则使工作历史更加干净。
    • merge的优点：没有触碰merge前的所有历史，而rebase修改了历史。那么多人协作若要回滚，merge更加有迹可循，知道是哪个分支的哪次commit造成的修改

    📌rebase需要解决过去历史中的conflict才可成功进行。

    • 黄金准则：不要rebase一个和别人共同工作的branch

  • commit维度的合并：

    上面的merge和rebase都是以分支为维度进行合并，但是可能会需要仅合并某条分支上的某几个commit，比如下面的b分支，只想将b分支的4号commit合并到a分支，产生新的commit 4’。此时可以利用git cherry-pick 4号commit。此外cherry-pick还提供了几种操作。

  • 合并多个commit：如左图，执行下述命令，就会将b分支的4号和8号合并到a分支。（合并后的commit号是新的）

    git cherry-pick 4号commit 8号commit

  • 合并左开右闭区间的commit：b分支的6、8合并到a分支

    git cherry-pick 4号commit..8号commit

  • 合并闭区间的commit：b分支的4、6、8合并到a分支

    git cherry-pick 4号commit^..8号commit

用cherry-pick合并的过程也不可避免会有合并冲突，可以有两种解决方案：

  1. 解决冲突：代码冲突解决后，git add <file>+git cherry-pick --continue
  2. 放弃合并，回滚：git cherry-pick --abort

  • 在不同分支间切来切去：

    比如正在dev分支上做修改，还没有修改完成，就被叫上去修改feature分支的内容了，这时又不能将dev上修改的内容直接提交。那么这个文件如果已经被add（可被追踪），就需要先执行git stash，表示将修改的内容进行缓存（压栈），且修改内容移除。之后便可跳到feature分支。

    当把feature分支的工作完成后，回到dev分支时，可以执行git stash pop，那么刚刚所作的修改就又回来了。

    📌git stash和git stash save是一样的。

远程仓库

  • git remote add origin https://github.com/username/repo-name.git：

    将github上的远程仓库绑定，以origin为名在本地标记，相当于一个新分支。添加好远程仓库后，就可以用git remote -v查看有哪些远程仓库的标记。

    若要解除某个remote的绑定，比如origin，则：git remote rm origin

  • git branch -M main：因为人权运动，github上不再有master分支，需改名为main

  • git push -u origin main：将本地main分支的内容push到远程仓库（origin）的main/master分支。只有第一次推送时才需要-u参数。

📌上面是将本地的代码第一次提交到远程仓库的步骤。如果是将已有的远程仓库克隆下来再修改，git clone自动会让文件夹为仓库且有名为origin的远程仓库标记，直接push/pull即可。

  1. origin的含义：

    origin表示远程仓库在本地的一个指针，通过此指针对远程仓库进行push/pull，也就是origin实际上指向origin/main这个远程分支。

    github指导用户添加远程仓库时，命名为origin，并且执行git clone的时候其实git自动将clone的远程仓库指针命名为origin。

    git merge origin master并不是直接将远程仓库代码合并到本地的master，而是可以感性地理解为将本地的origin分支合并到本地的master。

    但是，对于github上开源项目的维护，一般并不是直接提交到origin/master，因为没有权限。即使有权限，直接提交到origin/master也是一项危险的操作。所以往往会创建一个新remote，并在本地建立一个新分支，详情在后面的“多人协作”。

  2. pull和fetch的区别：

    • fetch：仅下载远程repo，即更新origin/master（或其他remote）的最新commit
    • pull：下载远程repo并合并，即pull=fetch+merge
      相比之下，fetch是一个更为保险的操作。

  3. push时的合并冲突：

    在push的时候可能也会出现合并冲突。因为在本地修改的时候，可能远程仓库已经被其他人提交了更改，这时要尽可能保持与最新版本代码的一致。需要先pull，再修改，再push（要求push的必须是已经领先于远程的）

    那么为了避免上述合并冲突情况的发生，所以一般建议在修改前更新本地代码：git branch -u origin/master

  4. fork仓拉取：

    • 自己是主仓，拉取fork仓的内容：

      在提交到远程仓库后，该仓库作为主仓，可被其他用户来fork，这时在拉取的时候既可以拉取自己的主仓，也可以拉取其他人的fork仓。可以直接按他人的仓库地址新建remote，拉取即可。

    • 自己是fork仓，拉取主仓的内容：

      自己在fork时，仓库除了有默认的remote origin外，还有一个remote: upstream，来绑定上游仓库，主仓就是自己的上游仓库。

      此时的git pull或git pull origin master是将远程的fork仓拉到本地，而非上游仓库，若要将上游仓库拉到本地，需要用git pull upstream master。

      若要同步远程的fork仓，将拉取后的内容合并到仓库：git merge upstream/master

      📌如果上游仓库的地址出现了变更，比如从ssh://变成https://，可以：
      git remote remove upstream ssh://...移除上游仓库地址，再
      git remote add upstream https://...重新添加正确地址。

  5. 多人协作规范（以github为例）：

    对于一个大型开源项目而言，前面说到直接push到origin/master不现实而且风险比较大。 如果要对大型项目进行更改，并且提交到上游：

    1. git clone 主仓，同时fork主仓x，得到自己的仓y；

    2. 添加新remote，并且禁用remote origin的push权限：

    3. 在本地开一个新branch，在此branch中而非master分支去修改代码；

    4. 修改后commit，再git push 新remote名 本地新branch名推到y仓；

    5. 命令行出现提交PR的网页链接，进入页面，描述问题后，提交PR

    6. 等待项目管理者code review，管理者可能会提出修改意见，此时根据意见再在本地修改。修改完成后仍执行git push 新remote名 本地新branch名，已开启的PR也会更新。如果管理者认可了修改，会在github上跑该项目预先的测试用例（CITest, Continuous Integration，持续集成测试），一般只有全部测试通过，才会被merge进主仓x。

    📌太极图形的Github开发建议：
    1. 不要在master分支直接进行操作，先开个branch
    2. 一个PR尽量只包含一个commit，清楚的写明干了啥（一件事）
    3. 提交PR的人应该优先考虑如何让review变得简单且快速
    “为什么要提这个PR，这个PR干了什么，怎么干的，还剩什么没干”

标签
